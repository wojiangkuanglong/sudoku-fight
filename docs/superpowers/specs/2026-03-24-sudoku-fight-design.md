# 双人实时对战数独 — 设计说明

## 目标

两名玩家在同一房间内同步解答**同一道标准 9×9 数独**；**先正确完成者胜**。对战亮点为**道具**：由服务端校验与广播，对对手产生限时或回溯类干扰。

## 架构

- **Monorepo（pnpm）**：`packages/shared`（类型与数独规则）、`packages/server`（Socket 实时与房间状态机）、`packages/web`（React + Vite 客户端）。
- **实时通道**：Socket.IO，房间以短码标识；**游戏状态以服务端为权威**（落子、胜负、道具效果、冷却）。

## 核心规则

- 开局下发：`givens`（题目，0 表示空）与 `solution`（仅服务端持有完整解；客户端只收到自己的格子与公开信息）。
- 玩家仅能修改非给定格；填入 1–9；清空为 0。
- **胜利条件**：玩家当前网格与 `solution` 完全一致。
- **道具**（首版，均可扩展）：
  1. **遮行**：指定一行，对手在约 30 秒内该行在 UI 上被遮挡（服务端记录 `rowBlindUntil[row]`）。
  2. **撤销三格**：从对手**最近用户填写**历史中弹出最多 3 笔（不撤销给定数字）。
  3. **冻结输入**：对手约 8 秒内不可提交填数（服务端拒绝 `cell:set`）。

每种道具每局有**使用次数上限**与**冷却时间**，由服务端强制执行。

## 数据流（摘要）

1. 玩家 A `createRoom` → 返回 `roomId`。
2. 玩家 B `joinRoom(roomId)` → 双方收到成员列表。
3. 双方 `playerReady` → 均 ready 后服务端 `game:started`（同一题目）。
4. `cell:set` / `item:use` → 服务端校验 → `state:patch` 或错误原因广播。

## 错误处理

- 非法房间、未满员开局、冷却中道具、冻结中填数：返回 `error` 事件带可读消息。

## 测试策略（后续）

- `shared`：纯函数单测（完成判定、历史撤销语义）。
- `server`：房间状态机单测（可抽离为纯函数 + mock 时钟）。
